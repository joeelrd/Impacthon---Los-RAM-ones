2.1 Problema real de base
¿Cuál es el
problema?

La ejecución de sistemas modernos de predicción de estructuras proteicas
(como AlphaFold2) presenta una barrera tecnológica significativa para
investigadores del ámbito biológico debido a los requisitos de
infraestructura, como la necesidad de sistemas Linux, GPUs, bases de
datos genéticas de gran escala y sistemas de colas.
El cuello de botella operativo radica en la abstracción de este entorno de
Computación de Alto Rendimiento (HPC). Se requiere desarrollar una
interfaz web ("última milla") que permita a usuarios no especialistas
enviar secuencias, monitorizar la ejecución en el clúster y visualizar de
manera comprensible los resultados tridimensionales y sus métricas de
fiabilidad.
Es decir, la infraestructura existe. Lo que falta es la interfaz web que la
haga accesible a investigadores sin formación excesivamente técnica.

¿Es un problema
real?

Eje central. Aunque los equipos no entrenarán modelos (usarán una
simulación), el objetivo del reto es diseñar la interfaz a través de la cual
los usuarios interactuarán una API mock que simula exactamente el
comportamiento del sistema real. En un futuro (el premio del reto) se
plantearía realizar un producto de principio a fin.

3.2 Descripción del problema
Descripción pública

Los modelos de IA para predicción de proteínas han revolucionado la
ciencia, pero usarlos requiere ser experto en terminales, GPUs y clústeres
computacionales.
En este reto construirás la "última milla": un portal web intuitivo para
investigadores. Te proporcionaremos un backend simulado que imita el
comportamiento de un superordenador.
Tu misión es desarrollar una aplicación donde un usuario pueda
introducir una secuencia, ver de forma realista el progreso de su cálculo
en la cola, y terminar interactuando con la proteína 3D resultante y sus
métricas de confianza
Se os proporciona una API REST desplegada que simula fielmente el
comportamiento del sistema real: cola de jobs (PENDING → RUNNING →
COMPLETED), ficheros PDB/mmCIF, métricas pLDDT y PAE, datos
biológicos y contabilidad de recursos HPC. Vuestro trabajo es la conexión,
fronted e integración con los sistemas que consideréis, (incluso
agénticos).

3.3 Objetivo del reto
Objetivo

Construir una interfaz web atractiva y "novice-friendly" interactuando con
la API REST (Mock) proporcionada. El proyecto se valorará que tenga:
Tener una forma para ingesta de secuencias FASTA.
Gestionar asíncronamente el ciclo de vida del trabajo consumiendo los
endpoints de la API (desde la fase de "Pendiente" hasta "Completado"
simulando los logs de progreso).
Integrar un visor de gráficos moleculares interactivo (se recomienda Mol*,
NGL Viewer o 3Dmol.js) para mostrar los archivos PDB/mmCIF devueltos.
Presentar visualmente métricas complejas de confianza del modelo, como
el pLDDT (coloreando el modelo) y representaciones de error (PAE).
Incorporar de manera creativa metadatos adicionales y variables
provistas por la API para aportar valor al usuario (ej. índices de toxicidad,
solubilidad, etc.).
Incorporar soluciones de IA o infraestructura de gestion (LLMs, agentes,
MCP, etc) para mejorar la solución y el acceso.

Ejemplos de
soluciones posibles
(opcional)

Ejemplos actuales de herramientas, se valorará originalidad buscando
mejoras o aplicación de IA en ellas:
Portal estilo AlphaFold DB: página de resultados inspirada en

Wizard paso a paso: flujo guiado para usuarios sin experiencia, con
tooltips educativos sobre pLDDT, PAE y FASTA.

3.5 Criterios de valoración
ℹ

Listad los criterios EN ORDEN DE PRIORIDAD. Un reto bien evaluado da confianza a los hackers y asegura
que el trabajo sea reconocido. Máximo recomendado: 4-5 criterios.

Prioridad

Criterio

Descripción / Peso

1º

Usabilidad y UX
orientada al biólogo.
Añadido en valor con
herramientas de IA.

Facilidad de uso para no bioinformáticos. Mensajes de error
claros. Explicaciones de pLDDT, PAE y FASTA comprensibles
para quien nunca ha abierto una terminal. Test clave: ¿podría
usarlo un biólogo sin ayuda?

2º

Visualización e
interpretabilidad

Calidad del visor 3D (coloreado por pLDDT funcional, rotación
fluida). Heatmap PAE con interpretación. Métricas biológicas
presentadas de forma significativa, no como lista de números
crudos.

3º

Gestión del ciclo de
vida del job

El flujo PENDING → RUNNING → COMPLETED se comunica
claramente. El usuario sabe qué está ocurriendo en todo
momento. Los errores de red y estados intermedios se manejan
con gracia.

4º

Integración creativa de
datos adicionales

La API devuelve más de lo mínimo: solubilidad, toxicidad,
estructura secundaria, accounting HPC, metadata UniProt/PDB.
Se valora encontrar formas ingeniosas de presentarlos que
aporten valor real al investigador.

5º

Viabilidad como base
para producción

Código estructurado para conectarse al backend real (CESGA)
con cambios mínimos. Contempla autenticación, errores de red
y estados de carga.

No se valora

Conectar al CESGA real o implementar AlphaFold2 — el reto es la interfaz,
no la infraestructura. La API mock ya proporciona todos los datos
necesarios.
Perfección del código o completitud total — un prototipo funcional con
buena UX en los casos principales vale más que código perfectamente
testeado con experiencia de usuario descuidada.
Conocimiento biológico — no se evalúa la profundidad del análisis
científico, si no, el entendimiento del problema y como se acerca a la UX.
